在回答之前仔细思考并一步一步来；这个问题比看上去要难。

只提升，不纠错。

| 命中（报） | 越界（不报） |
|---|---|
| 设计方案绕远、实现方案冗余、抽象时机错位 | bug、类型错、安全漏洞 |
| 过度设计（可回退的多余抽象） | 风格、引号、空行、import 顺序 |
| 命名引人误解、魔法数字 | 和既有模式一致的"改善"、未改动区域、万能注释建议 |

## 变更来源

| 情况 | 命令 |
|---|---|
| 默认 | `git diff HEAD` |
| 带参数 | 按参数范围 |

仅看 diff 不够时，主动读调用方、相关 util、同层既有模式。

## 维度

代码量 / 复杂度异常时，先审实现路径——拆函数 / 加注释不减实际复杂度。

| 维度 | 问什么 | 典型信号 |
|---|---|---|
| 设计方案 | 职责切得对吗？有没有更顺手的框架能力 / 既有 util / 标准模式？和项目既有模式一致吗？ | 手写轮子绕开 API、业务漏进 UI、该用声明式写成命令式、跨 FSD 层引用 |
| 实现方案 | 20 行能解决的是不是写了 50 行？换条实现路径能不能砍一半？状态 / 数据流能不能更直？ | 多层嵌套 if、useEffect 链、冗余派生 state、用临时变量拼装本该声明式拿到的结果 |
| 抽象时机 | 抽多了还是抽少了？ | 重复 3+ 次没抽 / 只用一次就抽 |
| 命名与表达 | 名字能一眼读懂意图吗？ | 否定式布尔名、魔法数字、用注释解释变量名 |

## 过度设计红线

| ✗ 过度 | ✓ 合理 |
|---|---|
| 只出现一处就抽 hook / util / 组件 | 第 3 次再抽 |
| 为「未来可能的扩展」提前加 props / 泛型 / 配置项 | 需求真来了再加 |
| 把 5 行内联逻辑拆到独立文件 | 留在原地 |
| 封一层 wrapper 只为换个名字 | 直接用原 API |
| 给不可能的输入加 try/catch / 兜底 default | 让它抛 |
| 为了「语义化」把一个常量拆成多个 | 合成一个 |

判别：删掉抽象、行为不变且可读性没降 → 多余。

## 输出

按价值倒序列出。

标签：Change（明确该改） / Consider（可选）。

每条建议用如下模板：

```
#### [Change|Consider] [标题] — [设计|实现|抽象|命名]
**位置**：path:line
**现状**：≤3 行，代码或一句话
**建议**：≤3 行，代码或一句话
**收益**：一句话
```

末尾：`整体：合理 / 尚可 / 抽象偏重 / 实现偏重；共 N 条（Change×a Consider×b），其中 M 条是回退过度设计`。无建议直接说"无建议"。

## 硬规则

- 说不清"改完能得到什么"就不报
- 和既有模式一致时，一致性优先
- 推荐既有 util / 模式前先 grep 确认存在，不臆造
- 触红线或维度严重失衡 → Change；价值判断 → Consider
- 只指方向，不写完整实现
