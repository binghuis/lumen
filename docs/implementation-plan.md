# 实现方案与架构决策(B站优先)

记录流明 AI VTuber 从"对标调研"收敛出的关键技术决策与 why,防止后续回滚到已被否决的方向。结论基于:Neuro-sama 等成功案例的实证、Open-LLM-VTuber / kimjammer-Neuro 的代码实测、以及 B站/抖音 的推流与合规政策核实。

## 决策一:首发平台 = B站,不是抖音

README 原定推流抖音;调研后改为 **B站 优先验证**。四条证据全部指向 B站。

| 维度 | B站 | 抖音 |
|---|---|---|
| 弹幕接入 | 开放(WebSocket / blivedm),开源已支持 | 无官方公开 API,需逆向,不稳 |
| 纯 AI 无人合规 | 基本可行(仅需 AI 标注),整治不针对娱乐 VTuber | 明确违规,要求真人实时值守 |
| 平台文化 | VTuber 老家,有对口观众与分区 | 带货为主,陪聊 AI 无成功先例 |
| 已验证案例 | Neuro-sama、AI-YinMei(吟美)均在 B站 | 无 |
| 变现天花板 | 打赏(舰长/SC/电池)真实但盘子小 | 流量与钱大得多 |

**Why 防回滚**:抖音的大流量绑定了三道墙——带货式合规枷锁、无弹幕 API、纯 AI 无人直播违规——三者都正面顶撞"纯 AI 陪聊 + 触发打赏"的 MVP 命题。抖音留作 B站 验证成功后、愿意额外承担真人值守+弹幕逆向+带货合规时的二期目标。

## 决策二:形态 = 2D Live2D 起步,UE5 后置为画面升级项

否决 README 原定的"UE5 + Pixel Streaming + 云 GPU"作为 MVP 前置。

**Why**:产品价值在对话闭环(延迟 + 人格 + 触发节奏),不在渲染保真度——Neuro-sama 用简单 2D 模型成为 Twitch 订阅第一即为实证。UE5 路线把复杂度压在"渲染"上,额外背上云 GPU 常驻成本、面部 rig、Pixel Streaming 稳定性、hero 美术关键路径,而这些都不直接产出要验的 liveness。2D/VRM 上"LLM 意图→动作/表情"的导演层逻辑同构,验证成本低一个数量级。UE5 当成画面升级项,不作 MVP 依赖。

## 决策三:技术栈选择

> 下表为**实际落地**的选型;与最初计划的最大偏差(没用 Open-LLM-VTuber fork)见「实现进展」。

| 层 | 实际选择 | Why / 状态 |
|---|---|---|
| 编排 | 自建轻量直连管线(`backend/bili_live/`) | MVP 验证更快;Open-LLM-VTuber 留作上 Live2D/前端时参考 |
| 弹幕接入 | blivedm 直接用(web 模式) | B站 弹幕开放,填 room_id(+SESSDATA)即用 |
| LLM | 火山豆包 Ark(OpenAI 兼容) | 区域内低延迟、中文好、便宜、零代码 |
| TTS | 火山 SeedTTS 2.0(v3 非流式)+ edge-tts 兜底 | 已通;**流式低延迟 + 声音复刻统一音色 = 待办** |
| 形象 | Live2D Cubism(自购/委托,商用授权) | **未做**;B站 VTuber 标准,轻、proven |
| 采集/推流 | 本机 BlackHole + 多输出 + OBS 采音频 | 语音进 OBS 已通;**B站 rtmp 推流受 5000 粉门槛阻塞,见开放问题** |
| 部署 | Mac 本机 + 托管 API,无需 GPU | 2D + 托管 LLM/TTS,本机只跑轻活 |

**关键认知**:LLM 人格主要由 system prompt + 角色设定 + 少量微调决定,换基座模型解决不了"无聊"。

## 决策四:自建 vs 借开源 的边界

判据(满足任一才自建,否则借开源):①是差异化;②开源没有且承重/担责;③开源设计顶撞核心 KPI。

**自建(按优先级)**:

| 模块 | 判据 | Why 自建更好 |
|---|---|---|
| 导演层 + "何时说话"触发节奏 | ① | 核心增量;开源触发逻辑是通用玩具,越 specific 越赢 |
| 角色人格 + 一致性记忆 | ① | 护城河;开源记忆通用/残缺(Open-LLM-VTuber 长期记忆已下线) |
| 内容安全双向护栏 | ② | 开源为零且是法律担责项,不可外包 |
| 流式 TTS 低延迟管线 | ③ | 开源是整句文件式契约,顶撞"超速回应"KPI,需换内核 |
| 推流/采集无人化稳定性 | ② | 开源靠人肉 OBS;24h 长稳需 headless 渲染→编码→rtmp+监控(MVP 后) |

**借开源(造了即浪费)**:B站弹幕协议(blivedm)、Live2D 渲染+音量驱动口型、LLM/TTS provider 适配层、弹幕背压队列(ProxyMessageQueue)、ASR/VAD。

## 决策五:合规约束(B站,全国性国标)

- 画面标注"虚拟主播/AI生成"(≥1/10 区域);开播前 10 秒语音+字幕表明 AI 身份。
- 双向敏感词过滤:入(弹幕)+出(LLM 回复)各一道,避免被弹幕诱导导致封播;预算够再加云内容安全 API 语义审核。
- **Why**:国标《人工智能生成合成内容标识办法》对所有平台适用;B站 整治压力虽小于带货平台,但敏感内容封播风险与账号主体担责不可裸奔。

## 实现进展(截至 2026-06-20)

后端落在 `backend/bili_live/`(uv 项目,Python 3.11)。**注意与决策三的偏差**:MVP 没用 Open-LLM-VTuber fork,而是自建了一条更轻的直连管线(弹幕→豆包→火山TTS→本机播放),验证更快;Open-LLM-VTuber 仍是后续要上 Live2D/前端时可借的参考。以下均用真实服务验证过:

- **弹幕接入**:blivedm,web 模式(`receiver.py`)。实测未登录也能收弹幕文本,只是用户名打码;SESSDATA 可从浏览器自动读取。
- **大脑**:火山豆包 Ark(`brain.py`,OpenAI 兼容)。人格「流明」+ 最近若干轮短期记忆(`BRAIN_HISTORY_TURNS`)。Ark 客户端 `trust_env=False`,避免境内请求走本机翻墙代理。
- **TTS**:火山 SeedTTS 2.0 **流式**(`voice.py`,ffplay 边出边播)+ 豆包按句流式(`brain.py`),首句先开口;edge-tts 兜底。接口要点见模块 README「火山 TTS 接口要点」。
- **编排**:`live_vtuber.py`,单飞 + 付费事件(礼物/SC/上舰)优先队列,普通弹幕/免费礼物满则丢弃保新鲜。
- **mode A**:`auto_reply.py`(读到→回发弹幕)是测试/辅助形态,非产品主形态。
- **音频上流(本机 macOS)**:BlackHole + 多输出设备「Lumen上流」+ OBS 采集,已实测语音进 OBS。配置见模块 README「音频上流」。

**尚未落地(已决策、未实现)**:
- 内容安全双向护栏 + AI生成标注(决策五)。
- 导演层:LLM 输出情绪/动作标签 + 触发节奏精化(现仅"何时说话"的优先队列)。
- Live2D 形象 + 音量驱动口型。
- 火山声音复刻统一音色(流式低延迟管线已做)。
- B站 正式推流上线(受 5000 粉门槛阻塞,见开放问题)。

凭据/配置全在 `backend/bili_live/.env`(gitignore);变量与各接口要点见该模块 README。

## 已知选型风险

- **Open-LLM-VTuber v1 已冻结、官方将重写 v2 且不兼容**:pin v1 自维护;预期导演层长大到与通用框架分叉时,把编排内核收归自建。
- **框架自带 Live2D 样本模型商用受限**(企业年销 >1000 万日元须 Live2D 书面授权,协作角色禁商用):正式上线必须换有商用授权的模型。
- **抖音弹幕源无官方 API**:若二期上抖音,弹幕获取有稳定性与合规风险。
- **B站 推流路径在「Mac + 低粉」下堵死(开放问题,2026-06 确认)**:
  - OBS/第三方**推流码需粉丝 ≥5000**(B站规则,开播设置页截图确认)。
  - **B站直播姬只有 Windows 版,Mac 装不了** → Mac + 低粉账号无法用 OBS/直播姬把 Live2D+语音合成推上 B站。
  - 手机扫码开播无门槛,但**不给推流码**,OBS 的画面/音频合成进不去流。
  - 互动玩法**身份码**(open_live 弹幕用)≠ 推流码,两者无关,别混。
  - 关键:**推流码统一受 ≥5000 粉限制,OBS / ffmpeg / 云推流都需要它**(ffmpeg/云直推并非绕过手段,纠正早前误判)。**唯一不需要推流码、无粉丝门槛的 PC 推流是 B站直播姬,但它仅 Windows**;手机扫码开播也无门槛,但画面/合成进不去。
  - 待解候选:借/装一台 Windows 跑直播姬(无 5000 门槛)/ 涨到 5000 粉后 Mac 用 OBS / MVP 先手机开播 + 占位画面只验内容。
